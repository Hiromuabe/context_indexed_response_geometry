# Assumptions

1. The repository's exact existing checkpoint is `Qwen/Qwen2.5-Math-1.5B` (base), as declared by `configs/basepoint_main.json`, `configs/trajectory_generation.json`, and the root README. The revision remains configurable and resolved metadata is recorded at runtime.
2. Existing greedy GSM8K trajectory JSONL files are the source of prefixes. The pipeline does not filter on correctness and uses one prefix per original problem by default.
3. Generated-token positions are divided into three deterministic progress strata. Within each stratum, a seeded choice provides early/middle/late balance.
4. Candidate-selection prefixes form a problem-disjoint group and are the only prefixes used to select the common token vocabulary. Auxiliary prefixes estimate token main effects and outlier directions; they are not candidate selectors.
5. The first available hidden state is represented by transformer block 0 output. Qwen's embedding output is not injected into a transformer-block continuation, so it is excluded from Experiment 2.
6. Content excludes absolute position 0 (BOS/attention sink) before results are inspected. Additional sink positions can only be supplied in the config before evaluation.
7. The good-match threshold is the configured quantile of development-prefix match distances. The threshold is frozen and then applied to evaluation prefixes.
8. The functional primary layer starts at block 16, matching the existing repository. Development results may select another layer/rank only by writing a new immutable config before test evaluation.
9. Smoke/pilot/full counts refer to total common candidates, split into disjoint calibration M and analysis V. Each of four folds partitions V into 3/4 training and 1/4 evaluation tokens.
10. GPU-dependent smoke and pilot results are not fabricated when the local machine lacks the trajectory files, Transformers, or CUDA; CPU numerical tests remain runnable.
11. Rank-0 output-distance stability is accepted when the range across configured development calibration sizes is at most 10% of the largest mean distance. This tolerance is fixed in YAML before evaluation.
12. The older `gsm8k_trajectories_smoke.jsonl` contains only eight problems and cannot satisfy the specified 32-prefix smoke pool. A response-subspace-specific deterministic smoke trajectory artifact generates up to 48 problems, leaving eligibility headroom while retaining a 32-problem analyzed pool.
13. A prediction match is not considered valid merely because it falls in a development-set quantile. The frozen absolute ceiling is normalized JS <= 0.25 (raw JS <= 0.25 log 2), and the matched median must improve on the random-pair median by at least 25%. These criteria are config-driven and fixed before evaluation.
14. “High-probability candidate” means that the forced token ranks in the top 128 or top 256 among valid tokenizer vocabulary IDs for that prefix. Raw probability quantiles over the deliberately broad common-candidate set are not used.
15. Rank-0 calibration-size stability is measured on development cells, while the reported functional score is measured on test cells. The full-M development condition is recomputed twice on exactly the same cells and must agree within the configured anchor tolerance; test and development means are labeled separately.
16. Permutation p-values are suppressed when fewer than 80% of primary-layer prefixes lie in length/progress strata containing at least two exchangeable prefixes. Recentring correctness remains a separate implementation gate.
17. Gate 7 is split: 7a tests positive Local recovery and Local > Global without depending on Matching quality; 7b additionally requires valid Matching and tests Local > Matched. Only 7b supports the “beyond the current prediction” functional claim.
18. Pilot Matched identifiability is audited before hidden-state branching on the full next-token prefix pool. Eligible donors must be from another problem, share top-1 prediction and fixed length/progress bins, maximize top-5 then top-20 overlap, and finally minimize full-tokenizer-vocabulary NJS. Pilot requires at least 32 good pairs, at least 80% matched-query coverage, NJS <= 0.25, and the preconfigured random-distance improvement; otherwise Matched is automatically treated as unavailable while Global/Content analyses continue.
